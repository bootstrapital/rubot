Rubot

Workflow Engineering For Real Business Processes.

Rubot is a code-first framework for building governed workflows where models can help with judgment, but the surrounding system still has explicit state, procedure, approvals, and long-term ownership.

Rubot as an ops backend

There is a growing class of software that sits between a simple script and a fully custom product surface. It is usually operational in nature: review flows, routing, approvals, intake, case handling, compliance checks, escalations, or financial procedures. This work is often too important to leave in spreadsheets, ticket queues, and ad hoc scripts, but too specific to receive the same treatment as a core customer-facing feature.

Large models make this category more interesting, but they do not remove the hard part. The hard part is not generating an answer. The hard part is building a system that can gather facts, apply judgment, stop for review, resume later, take bounded action, and leave behind a durable explanation of what happened.

Think of Rubot as an ops backend for this kind of work. A product UI can submit work into it. An API can submit work into it. A webhook or scheduled process can do the same. In each case, the submission becomes a durable operational run that can gather facts, apply judgment, branch under policy, pause for approval, and then take bounded action.

The closest analogy is probably dbt. dbt did not invent SQL or the warehouse. It gave analytics engineering a shaped layer for turning raw data work into something more durable, reviewable, and owned. Rubot is aiming at a similar role for operational systems. It does not replace the host application, background jobs, or external APIs. It provides a structured runtime where incoming requests, signals, and cases can be turned into governed procedures with explicit actions, judgment boundaries, approvals, and history.

That is the problem Rubot is trying to address. It treats inference as one component inside a broader operational system rather than the category itself.

Code-first is pragmatic

Rubot is code-first in a very specific sense. It is not a prompt layer, a hosted workflow builder, or a visual automation tool. It is a framework for writing operational capabilities directly in application code, with enough structure that the resulting system can still be reviewed, governed, and maintained over time.

This approach is more practical now because code generation has lowered the cost of bespoke workflow software. Teams can describe a process and have large parts of the implementation drafted for them. That makes code-first systems much more approachable than they were even a few years ago.

But the answer is not to abandon code. Business processes become complex in ways that are hard to express cleanly in prompts or GUIs alone. They involve branching logic, exception handling, policy boundaries, integrations, state transitions, and human review. Code is still the most expressive and malleable medium for that level of complexity, as long as the framework gives it a durable shape.

In other words, code generation makes bespoke workflow software easier to produce. A code-first framework is what makes that software easier to keep.

Workflow Engineering

A useful way to think about this is as workflow engineering: the practice of building operational software that coordinates systems, humans, policy, state, and AI where useful. The unit of work is an operational capability, not a markdown page or prompt.

In that sense, the question is not whether a model can classify, summarize, recommend, or draft. Models can often do those things well enough to be useful. The question is whether the resulting workflow remains structured enough to trust, review, govern, and maintain once it begins touching real business processes.

The dbt analogy is useful here for another reason. Analytics Engineering helped create a path for analytically minded people to grow into a more code-native discipline without severing their connection to the business problems they understood best. Workflow Engineering creates a similar opportunity for strong operations practitioners. The people closest to the process often understand the real failure modes, policy edges, and handoffs better than anyone else. As software generation gets cheaper, there is an opportunity to help that talent move up the stack into more explicit system design and workflow ownership.

Better models make software generation cheaper. They do not remove the need for architecture, approvals, replay, regression control, and clear boundaries around side effects.

Why Ruby and Rails

Ruby is a good place for this because the framework can stay powerful without becoming unreadable. Workflow code still needs to look like application code that a team can understand. If the target architecture is legible, generated code has a much better chance of being worth keeping.

Rails matters because real operational systems usually need persistence, jobs, authentication, administration, and integration with the rest of an application. That makes Rails a strong host environment for governed workflow software rather than just a place to mount a demo.

The primitives

Rubot introduces a small set of primitives intended to keep these systems legible.

Tool

A tool is an explicit action boundary. It loads facts, calls an external system, or performs a concrete side effect.

Agent

An agent is a structured judgment boundary. This is where a model is useful for classification, extraction, summarization, or recommendation.

Workflow

A workflow defines procedure: ordered steps, branching, approval points, resumability, and state transitions.

Operation

An operation is the capability boundary. It is the unit a host application launches when a piece of operational work enters the system.

Run

A run is the durable execution record: events, outputs, checkpoints, approvals, replay relationships, and trace history.

The point of these boundaries is not abstraction for its own sake. The point is to stop a workflow from collapsing into a single opaque loop where prompt text, control flow, policy, and side effects are all mixed together.

What this enables

The practical outcome is semi-autonomous software that can operate with clear boundaries. A workflow can proceed on its own where policy allows, stop where review is required, and remain understandable to the humans who own it.

That makes this a good fit for systems where trust, reviewability, and maintainability matter as much as model capability itself. The interesting question is not whether a model can act. It is whether the resulting workflow still feels like real software.

Rubot is workflow infrastructure for governed operational software.
